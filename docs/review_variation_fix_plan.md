# 打谱复盘 · 变化图与胜率问题核对与修复方案

## 一、问题核对结果

### 1. 胜率「前几步没了」与顺序错乱

**可能原因 A：图表 X 轴尺度随当前分支变化**

- 当前实现：`maxTurn = _currentLine().length`。
- 在主战线时：`_currentLine()` = 主战线到结尾，长度为**主战线总手数**（如 200）。
- 在变化图时：`_currentLine()` = 当前路径 + 沿当前分支延续到结尾，长度为**当前分支总手数**（如 29+15=44）。
- 后果：在变化图时，图表 X 轴只有 0..44，而主战线胜率点的横坐标是 1..200，大部分会画到 x>1 的区间（被裁掉或看不见），看起来就像「前几步的胜率没了」或整条主战线错位。

**可能原因 B：主战线胜率键 `""` 被漏存**

- 持久化时对 `_winratesByBranch` 做 `if (entry.value.isEmpty) continue`，空的分支不写入。
- 若从未在主战线分析过，`_winratesByBranch[""]` 可能为空，保存时就不会写入 `""`，下次加载就没有主战线胜率。
- 若之前是旧格式（扁平 `{"1":0.5,...}`）加载到 `""`，之后只在变化图分析并保存，理论上仍会一起保存 `""`；只有在某种情况下内存里 `""` 被清掉或从未加入，才会丢。

**结论**：优先修复「图表 maxTurn 随分支变」导致的显示错乱；并保证主战线键 `""` 在加载/保存时不被误删。

---

### 2. 变化图「跑到棋局结尾、手数莫名其妙」

**原因：手数语义混用**

- 当前：`_currentTurn = _path.length`（路径节点数）。
- 在主战线：手数 = 主战线第几手，正确。
- 在变化图：路径 = 主战线前 29 手 + 变化图内 5 手 → `_currentTurn = 34`，界面显示「手数: 34」。
- 用户期望：主战线总手数固定（如「共 200 手」）；在变化图时应区分「分支点手数」与「变化图内第几手」，例如「第 29 手 · 变化 第 3 手」，而不是一个混在一起的 34。

**变化图链接出现位置**

- 当前：在任何「当前节点 `children.length > 1`」的地方都显示「变化图1」「变化图2」。
- 也就是说：**变化图链接出现在「发生分支的那一手」的笔记区**。若 SGF 里分支就在第 29 手，就会在第 29 手出现；若 SGF 里分支在结尾（例如官子多种下法），就会在结尾出现。
- 「跑到棋局结尾」很可能是：该 SGF 中**多子节点恰好出现在结尾**（例如根或最后一手有多个分支），而不是代码把「第 29 手」画到结尾。需要确认具体 SGF 结构。

**结论**：手数显示要区分「主战线手数」与「变化图内手数」；主战线总手数固定显示，不随进入变化图而变。

---

### 3. 变化图应在「发生的那一步」写入笔记、与用户笔记不冲突

**当前实现**

- 笔记区 = 当前节点的 `comment`（打谱笔记）+ 若 `children.length > 1` 则显示「变化图1」「变化图2」链接。
- 链接是 UI 上的入口，并没有「自动写入」到节点的 `comment` 里；用户编辑笔记只改 `comment`，和是否显示变化图链接不冲突。

**用户期望（理解）**

- 「变化图应该在变化图发生的那一步自动写入笔记」：应指**在该手（分支点）的笔记区里自动出现变化图入口**，而不是必须从别处进。当前已经是在「该手」的笔记区显示链接，符合「在该步出现」；若 SGF 分支点就在该手，就不会跑到别的手。
- 「用户编辑笔记和变化图同时保存不冲突」：当前编辑笔记只改 `comment`，保存 SGF 时一起写回；变化图是树结构（多子节点），不写进 `comment`。两者不冲突，只需保证保存时既写 `comment` 又写树结构即可（当前 `serializeSgf` 已包含）。

**结论**：逻辑上无需大改；重点是手数、maxTurn 修好后，「第几步」对用户清晰，避免误以为「跑到了结尾」。

---

### 4. 变化图手数独立、不影响棋盘总手数

**当前问题**

- 总手数没有固定展示：界面只显示 `_currentTurn`（= 路径长度），在变化图里会变成 34 这种「混合手数」。
- 没有「主战线共 N 手」的固定展示，也没有「变化图内第 k 手」的单独展示。

**期望**

- 主战线总手数固定：例如始终用 `_sgf!.mainLineNodes().length` 显示「共 200 手」。
- 在主战线：显示「手数: 29 / 200」或「第 29 手 / 共 200 手」。
- 在变化图：显示「第 29 手 · 变化 第 3 手」或「29 手 (变化 3/15)」，使分支点与变化图内手数独立、总手数不变。

## 二、修复方案（按优先级）

### 修复 1：图表 maxTurn 固定为主战线长度

- **改法**：胜率图 `maxTurn` 不再用 `_currentLine().length`，改为 **主战线总手数**：`_sgf!.mainLineNodes().length`（或只计有 move 的节点）。
- **效果**：无论当前在主战线还是变化图，X 轴都是 0..主战线长度，主战线胜率始终完整显示；变化图胜率点会落在其对应手数位置（分支点及之后），不会因「当前分支长度」把主战线挤没。

### 修复 2：手数展示区分主战线 / 变化图，总手数固定

- **主战线总手数**：  
  `int get _mainLineLength => _sgf?.mainLineNodes().length ?? 0;`（或只算有 `move` 的节点）。
- **在主战线时**：  
  显示「手数: $_currentTurn / $_mainLineLength」或「第 $_currentTurn 手 / 共 $_mainLineLength 手」。
- **在变化图时**：  
  - 分支点手数 = 进入变化图前的路径长度（即主战线上的第几手），例如 29。  
  - 变化图内手数 = 当前路径长度 - 分支点手数，例如 3。  
  - 显示例如：「第 29 手 · 变化 第 3 手」或「29 (变化 3/15)」，并保留「共 $_mainLineLength 手」。
- **效果**：总手数始终是主战线长度；变化图手数独立，不混成「34」这种数。

### 修复 3：胜率按「手数」对齐到主战线尺度（可选，与修复 1 配合）

- 变化图分支的胜率目前是按「当前分支内 turn」存的（1, 2, …, 当前分支长度）。
- 若希望变化图胜率在图表上落在「主战线第 29、30、31…」的位置，需要在存/取时用「主战线等价手数」：分支点手数 + 变化图内序号。这会影响 `_winratesByBranch` 的 key 语义和 _pathToBranchKey。若先只做修复 1+2，可暂不改这里，仅保证主战线胜率完整、手数清晰。

### 修复 4：保证主战线胜率键 `""` 不丢

- 加载：保持现有逻辑；旧格式继续映射到 `""`，新格式已有 `""` 则保留。
- 保存：若本次会话从未写入过 `""`，保存前不要覆盖或删除已有 `record.winrateJson` 中的主战线数据；或在合并写回时，若内存没有 `""` 则保留原 JSON 里 `""` 的内容（需要读出现有 winrateJson 再合并）。
- 这样即使只在变化图分析，也不会把原来的主战线胜率清掉。

### 修复 5：变化图链接与笔记的展示（核对即可）

- 当前：笔记区 = 当前节点 `comment` + 若有多子则显示「变化图1」「变化图2」链接；链接点击 = `_goToVariation(childIndex)`，进入对应分支。
- 确认：链接只在该节点有多个子节点时显示，即「变化图发生的那一步」；若实际 SGF 里分支在结尾，就会在结尾出现，这不是 bug，而是和 SGF 一致。若希望「只在分支点显示」且不依赖 SGF 多子位置，需要额外规则（例如只在该手是「第一次分叉」时显示），一般不需要。
- 用户编辑笔记与变化图同时保存：已不冲突，保存时 SGF 序列化同时写 `comment` 和树结构即可。

## 三、建议实施顺序

1. **先做修复 1（maxTurn）和修复 2（手数展示）**：解决「前几步胜率没了」「手数莫名其妙」「总手数不固定」的体感问题。
2. **再做修复 4（主战线胜率不丢）**：避免保存时误删主战线胜率。
3. 修复 3 视需求再做（是否要让变化图胜率按主战线手数对齐）。
4. 修复 5 以核对为主，确认 SGF 分支位置与「变化图在该步出现」一致即可。

## 五、补充修复：SGF 解析后规范化（主战线不被变化图顺序影响）

**现象**：打谱时变化图正常；退出再加载后「总手数莫名增加」「变化图没出现在发生的位置」。

**原因**：SGF 中若变化写在主续着前面（如 `(;A;B(;C);D)`，C 为变化、D 为主续着），解析会先遇到 `(;C)` 再遇到 `;D`，导致 `B.children = [C, D]`，`mainLineNodes()` 沿 `children[0]` 会走进 C，主战线变成 A→B→C…，总手数错、分支点错。

**修复**（已做）：在 `sgf_parser.dart` 的 `parse()` 中，`_parseTree` 之后调用 `_ensureMainLineFirst(root)`：
- 对每个节点，若有多个子节点，按「到叶子的路径长度」排序，将**最长分支**放在 `children[0]`；
- 这样 `mainLineNodes()` 始终沿最长分支 = 主战线，总手数 = 主战线长度，变化图只出现在支线、不影响主线；
- 变化图链接仍在「当前节点 `children.length > 1`」的笔记区显示，即分支发生的那一手。
